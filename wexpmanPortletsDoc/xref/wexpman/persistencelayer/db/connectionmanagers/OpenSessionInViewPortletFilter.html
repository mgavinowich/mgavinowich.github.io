23  /**
24   * 
25   * This class implements a filter which will open a database session when
26   * initiating the request and close it when the response has been generated. The
27   * session will be bound to the current thread. This filter constitutes an
28   * implementation of the open session in view pattern.
29   * 
30   * Given that the portlet specification (JSR 168) does not support portlet
31   * filters, this filter relies on the Apache Portals Bridges Portlet Filter
32   * library, which needs to be included with the project as well as properly
33   * configured in the portlet.xml (descriptor) file.
34   * 
35   * This class takes ideas from the OpenSessionInViewFilter class from Spring. It
36   * is designed to work with Spring and will use the session factory bean defined
37   * in Spring's context.
38   * 
39   * Portlet filters implemented with the Apache Portlet Filter library
40   * distinguish a process action filtering stage and a render filtering stage. As
41   * defined by the portlet life cycle, it can be assured that a process action
42   * will always be followed by a render request. Therefore, the process action
43   * filtering stage will not attempt to close the session, rather keep it open
44   * for the later rendering stage.
45   * 
46   * In the way filters are implemented by the library used, a given filter class
47   * will be instantiated only once per portlet, meaning that if two different
48   * users access the same portlet, the same filter instance will be used in both
49   * cases. This prevents from using member variables to keep state which depends
50   * on the user. It also bans using member variables to keep state which depends
51   * on the thread, because each user is serviced on a separate thread by the
52   * portal framework, therefore using a member variable would mean the same
53   * variable would be accessed by the different threads to update their
54   * thread-dependent information. For this reason, reference counts used by this
55   * portlet are stored in the PortletSession, in APPLICATION_SCOPE. Attributes
56   * stored under this scope are visible for all portlets only for a particular
57   * user (for practical purposes, this also means for a particular thread). These
58   * attributes survive between requests.
59   * 
60   * This filter will guarantee at most 1 session per request, per user, per
61   * portlet is employed. For a particular thread (i.e. user), the first filtering
62   * stage will open a session, associate it with the current thread, and
63   * increment the reference count of the session by one. If no other instance of
64   * the OpenSessionInViewPortletFilter is executed, once back from executing the
65   * rest of the filters in the chain, the reference count will be decremented
66   * and, if 0 (as it would be in this case), the session will be closed and
67   * unbound from the current thread. It would be expected that filter execution
68   * for different portlets for the same user (same thread) should not overlap,
69   * i.e. all filters for portlet 1 would be processed, then all filters for
70   * portlet 2, and so on. In the event there was overlap, and the
71   * OpenSessionInViewPortletFilter for portlet 2 was executed when the
72   * OpenSessionInViewPortletFilter has not been fully processed for portlet 1
73   * yet, the reference count would come into play (since it is the same
74   * user/thread which can see the APPLICATION_SCOPE attribute) and the same
75   * session would be used for both portlet 1 and portlet 2, effectively being
76   * closed once the reference count is 0. This also applies if for some reason
77   * two instances of this filter were applied to the same portlet, in which case
78   * execution would indeed be overlapped (although there is no reason for
79   * applying this filter twice to the same portlet).
80   * 
81   * If there is processing of an action, there will be a process action filtering
82   * stage for the portlet which is responding to the user action, plus as many
83   * render filtering stages as portlets are visible (as long as they use this
84   * filter). In this case, the session will be first opened in the process action
85   * filtering stage. Assuming no overlap in filter execution between portlets,
86   * the session will be closed in the render filtering stage for that same
87   * portlet.
88   * 
89   * @author Matias Alberto Gavinowich
90   * 
91   */
92  public abstract class OpenSessionInViewPortletFilter implements PortletFilter {

155 	 * 
156 	 * Applies the filter at the process action stage. Called by the portal
157 	 * framework each time there is an action request on a given portlet.
158 	 * 
159 	 * This method will open a new database session and attach it to the
160 	 * current thread (most likely case) or get the already attached session if
161 	 * there was one (unlikely, since process action is the very first stage
162 	 * executed from a given user action).
163 	 * 
164 	 * It will also increment the reference count on the session. Nevertheless,
165 	 * it would be expected that this is the only portlet processing an action
166 	 * and that the action processing for the portlet takes place before the
167 	 * rendering stage of any portlet (all this as defined by the portlet life
168 	 * cycle), therefore the most likely case is that when this method is
169 	 * finishing and decrements the reference count again, it will be back at 0.
170 	 * 
171 	 * This method will decrement the reference count when it ends execution,
172 	 * but will keep the database session open and bound to the current thread
173 	 * even if the reference count is 0 - which is what will likely happen -
174 	 * since it is expected that there will be at least one rendering stage (at
175 	 * least the one for this same portlet) for the same user (i.e. thread).
176 	 * Alternatively, this method could have been implemented so as to close the
177 	 * session, but then more sessions would be used overall, thus the approach
178 	 * chosen minimizes the amount of necessary sessions.
179 	 * 
180 	 * @param request
181 	 *            the {@link ActionRequest}
182 	 * @param response
183 	 *            the {@link ActionResponse}
184 	 * @param chain
185 	 *            the {@link PortletFilterChain} instance that allows access to
186 	 *            the chain of filters, in particular to invoke the next filter
187 	 *            in the chain, as required by the filter invoking scheme
188 	 * 
189 	 * @throws PortletException
190 	 *             if the filter could not be executed, this is defined by the
191 	 *             PortletFilter interface and it could actually only be thrown
192 	 *             if another filter down the chain throws it, since there is no
193 	 *             part of this method that throws such an exception.
194 	 * 
195 	 * @throws IOException
196 	 *             if there was a problem performing an I/O operation, this is
197 	 *             defined by the PortletFilter interface and it could actually
198 	 *             only be thrown if another filter down the chain throws it,
199 	 *             since there is no part of this method that throws such an
200 	 *             exception.
201 	 * 
202 	 * @see org.apache.portals.bridges.portletfilter.PortletFilter#processActionFilter
203 	 *      (javax.portlet.ActionRequest, javax.portlet.ActionResponse,
204 	 *      org.apache.portals.bridges.portletfilter.PortletFilterChain)
205 	 */
206 	@Override
207 	public void processActionFilter(ActionRequest request, ActionResponse response, PortletFilterChain chain)
208 			throws PortletException, IOException {
209 		
210 		logger
211 				.debug("Beginning execution of OpenSessionInViewPortletFilter process action filtering stage for portlet "
212 				+ this.portletConfig.getPortletName() + " in thread (meaning user) "
213 				+ Long.toString(Thread.currentThread().getId()) + " (filter instance id: " + this.toString() + ") ");		
214 
215 		beginUsingSession(request);
216 
217 		logger.debug("At OpenSessionInViewPortletFilter (filter instance id: " + this.toString()
218 				+ ") process action filtering stage, invoking next filter in chain ");
219 		
220 		chain.processActionFilter(request, response);		
221 		
222 		logger.debug("Resuming execution of OpenSessionInViewPortletFilter process action filtering stage for portlet "
223 				+ this.portletConfig.getPortletName() + " in thread (meaning user) "
224 				+ Long.toString(Thread.currentThread().getId()) + " (filter instance id: " + this.toString()
225 				+ "), back from next filter in chain ");		
226 		 		
227 		// keep session open because there will always be a render after a
228 		// process action
229 		endUsingSession(request, false); //changed to false because of issues with Db4o
230 		
231 		logger.debug("At OpenSessionInViewPortletFilter (filter instance id: " + this.toString()
232 				+ ") process action filtering stage, finished filtering ");		
233 	}

236 	 * 
237 	 * Applies the filter at the render stage. Called by the portal framework
238 	 * each time there is a render request on a given portlet.
239 	 * 
240 	 * This method will open a new database session and attach it to the
241 	 * current thread (most likely case) or get the already attached session if
242 	 * there was one (unlikely, since it would be expected that filter execution
243 	 * for different portlets for the same user - i.e. same thread - should not
244 	 * overlap, meaning all filters for portlet 1 would be processed, then all
245 	 * filters for portlet 2, and so on, so the reference count at the end of
246 	 * each portlet filtering would effectively be 0). In the event two
247 	 * instances of this filter are applied to the same portlet (though there is
248 	 * no reason to do so) or another filter is developed based on this
249 	 * implementation and the two of them are applied to the same portlet, the
250 	 * second filter executed would use the session already bound to the current
251 	 * thread.
252 	 * 
253 	 * It will also increment the reference count on the session when it begins
254 	 * using it and decrement it when it's done. If the reference count becomes
255 	 * 0 at this point (most likely), it will unbind the session from the
256 	 * current thread, and close it.
257 	 * 
258 	 * @param request
259 	 *            the {@link RenderRequest}

266 	 * 
267 	 * @throws PortletException
268 	 *             if the filter could not be executed, this is defined by the
269 	 *             PortletFilter interface and it could actually only be thrown
270 	 *             if another filter down the chain throws it, since there is no
271 	 *             part of this method that throws such an exception.
272 	 *

373 	 * {@link OpenSessionInViewPortletFilter#SESSION_REF_COUNT_ATTRIBUTE_NAME}
374 	 * 
375 	 * It is a protected method so subclasses can use it if this filter is
376 	 * customized.
377 	 * 
378 	 * @param request
379 	 *            the {@link PortletRequest} for this filtering stage, it can be
380 	 *            either an {@link ActionRequest} or a {@link RenderRequest}
381 	 *            depending on whether this method is being called from a
382 	 *            process action or a render filtering method.
383 	 * @return an Integer with the session reference count
384 	 */
385 	protected Integer getSessionRefCount(PortletRequest request) {
386 		PortletSession portletSession = getPortletSession(request);
387 		Integer refCount = (Integer) portletSession.getAttribute(SESSION_REF_COUNT_ATTRIBUTE_NAME,
388 				PortletSession.APPLICATION_SCOPE);
389 		if (refCount == null) {
390 			// if the reference count had not been set yet, it means it is 0
391 			refCount = new Integer(0);
392 		}
393 		return refCount;
394 	}
